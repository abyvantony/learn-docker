# Level 1: What is Docker? Containers explained simply

**Track:** Docker · **Level:** 1/12 · **Cycle:** 1 · **Difficulty:** `beginner`

📚 **Today's lesson** — published 2026-07-14

## TL;DR

Docker packages your app and everything it needs (code, libraries, system tools) into a single unit called a container. The container runs the same way on your laptop, your teammate's Mac, and your production server. No more 'works on my machine'.

## Real-world analogies

- A Docker container is a shipping container for software. Just as a real shipping container holds anything and stacks on any ship/train/truck, a Docker container holds any app and runs anywhere Docker is installed.
- A virtual machine is a whole house — full kitchen, bedroom, walls. A container is just a room — has what your app needs, shares the building (OS) with other rooms.
- Dockerfile = the recipe. Image = the frozen meal (built from the recipe). Container = the meal on your plate (running).

## Key concepts

### `Container vs VM`

VMs virtualize the hardware (each has its own OS). Containers virtualize the OS (share the host's kernel). Containers are smaller, faster, and start in milliseconds.

### `Image`

A read-only blueprint. Like a class in OOP. Built once, run many times.

### `Container`

A running instance of an image. Like an object instance. Has its own filesystem, network, processes.

### `Docker Hub`

Public registry of pre-built images. Like npm for Docker. `docker pull` downloads an image.

## Code with comments

Every line has a comment. Read it slowly.

```
# === Install Docker ===
# macOS: download Docker Desktop from docker.com
# Windows: Docker Desktop (WSL2 backend)
# Linux: sudo apt install docker.io && sudo usermod -aG docker $USER

# === Your first container ===
# Pull the official "hello-world" image
docker pull hello-world

# Run it — Docker downloads if not cached, then executes
docker run hello-world
# Output: "Hello from Docker! This message shows..."

# === What just happened? ===
# 1. Docker client contacted the Docker daemon
# 2. Daemon pulled the hello-world image from Docker Hub
# 3. Daemon created a new container from that image
# 4. Daemon ran the container, which printed the message
# 5. Container exited (it was a one-shot program)

# === See what's on your system ===
docker images          # All images you've pulled
docker ps -a           # All containers (running + stopped)
docker ps              # Only running containers
docker info            # Docker system info

# === Container lifecycle ===
docker run nginx                    # Run in foreground (terminal blocks)
docker run -d nginx                 # -d = detached (background)
docker ps                           # See it running
docker stop <container_id>          # Stop it
docker start <container_id>         # Start it again
docker restart <container_id>       # Restart
docker rm <container_id>            # Remove stopped container
docker rm -f <container_id>         # Force remove (even if running)

# === Naming containers (instead of random IDs) ===
docker run --name my-web -d nginx
# Now refer to it by name: docker stop my-web

# === Run a one-off command in a new container ===
docker run --rm alpine echo "Hello from Alpine Linux"
# --rm = remove container after it exits
# alpine = tiny Linux image (5MB)
# echo "..." = command to run

# === Interactive shell in a container ===
docker run -it --rm ubuntu bash
# -i = interactive (keep stdin open)
# -t = allocate a terminal
# You're now in a shell inside Ubuntu. type 'exit' to leave.

# === Get info about a running container ===
docker inspect <container_id>       # Full JSON details
docker logs <container_id>          # Its stdout/stderr
docker stats                       # Live CPU/memory for all running
docker top <container_id>          # Processes inside the container
```

## Try it yourself

1. Pull the `nginx` image. 2. Run it in the background with name `web`. 3. Visit http://localhost:80 — see the welcome page. 4. Stop it. 5. Start it again. 6. Remove it. Use `docker ps -a` at each step to see the state.

## Common pitfalls

- ⚠️ Confusing image and container. Image = template, container = running instance. One image → many containers.
- ⚠️ Forgetting `-d` (detached) — your terminal blocks until the container exits.
- ⚠️ Using `docker rm` on a running container — fails. Use `-f` to force, or `docker stop` first.
- ⚠️ Containers are not permanent. Remove `-rm` carefully — everything inside is gone.
